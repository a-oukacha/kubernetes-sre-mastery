# Lecture 15 - DaemonSet & node agents: one pod per node, and the blind spots

## Answers to the lab checkpoints
- **(0) A DaemonSet has no `replicas:` because it doesn't pick a number - it picks nodes**. Its desired count is *the number of nodes that match its node selector/affinity **and** that its pod can tolerate*. The DaemonSet controller maintains one pod per such node; add nodes, the count rises; remove or taint-out nodes, it falls.
- **(1)** One pod per `Ready`, schedulable node - so `DESIRED == CURRENT == READY == N` for the all-nodes agent. With no `nodeSelector` and no taints in play, every node matches.
- **(1b)** Never two on one node. The DaemonSet controller's invariant is *exactly one* pod per matching node; if it sees a node without its pod it creates one, and it never creates a second. Each pod's log prints its own `spec.nodeName` (injected via the downward API), so the heartbeats are distinct per node.
- **(2)** Two mechanisms, same controller. **New node: the DaemonSet controller watches the Node API; when a node appears that matches, it creates a pod for it - no edit to the DaemonSet, that's the whole point. Deleted pod:** `DESIRED` did **not** change (the node still matches), so the controller observed a matching node missing its pod and created a replacement. Same reconciliation loop you saw in lab 01, keyed on nodes instead of a replica count.
- **(3a) `DESIRED = 0`. No node carries `agentpool=observed`, so zero nodes match, so the DaemonSet wants zero pods. Nothing errors** - a DaemonSet matching no node is a perfectly healthy DaemonSet covering nothing. (This is Break-it A, and it's the quiet danger.)
- **(3b)** After labeling one node, `DESIRED = CURRENT = 1`, and the pod lands on exactly that labeled node (`$NODE1`). The plain `node-agent` is unaffected - it has no selector, so it still covers all nodes. Two DaemonSets, two coverage sets.
- **(4)** The pod already on `$NODE2` is **not** evicted - `NoSchedule` only blocks *new* placements; eviction of running pods is `NoExecute`. But once that pod is gone (you deleted it), no replacement lands, because the agent doesn't tolerate the taint. `$NODE2` becomes the blind node. The total `node-agent` count drops by one and `DESIRED` drops too - the controller no longer counts `$NODE2` as a node it can schedule onto.
- **(4b)** No `node-agent` returns to `$NODE2`. `DESIRED` dropped by one: DaemonSet `desiredNumberScheduled` excludes nodes the pod can't tolerate, so a missing-toleration node simply *leaves the denominator* - which is exactly why the gap is invisible unless you compare against `kubectl get nodes`.
- **(5)** `node-agent-tolerant` has a pod on every node, `$NODE2` included, because it carries `tolerations: dedicated=special:NoSchedule`. The two node lists (agent pods vs. all nodes) match. The single differentiating field is the toleration.
- **(B1) `DESIRED = 0`, `CURRENT = 0`, zero pods, no error, no event, no crash**. The selector matched nothing, so the controller correctly wants nothing. An agent that covers nothing looks identical in `kubectl get ds` to a brand-new healthy one - the failure is the *absence* of coverage, which monitoring on the agent itself can't catch (there's no pod to scrape).
- **(B2) The plain `node-agent` has no pod on `$NODE2`; `node-agent-tolerant` does. The one field that makes the difference is the toleration** for `dedicated=special:NoSchedule`. In production this is fluent-bit silently not shipping that node's logs.
- **(B3)** After repointing the selector to `agentpool=observed`, the subset agent returns to the labeled node(s); the tolerant agent's distinct-node count equals the cluster node count. Coverage restored, no blind spots.

## What just happened (under the hood)
A **DaemonSet** is a controller whose desired state is "one pod per node that matches." It does not own a replica count; it owns a *coverage relation* between pods and nodes.

1. The DaemonSet controller watches Nodes and Pods. For each node that matches the pod template's `nodeSelector`/`nodeAffinity` and whose taints the pod tolerates, it ensures exactly one owned pod exists. Node added -> create a pod. Node removed -> its pod is garbage. Pod deleted on a still-matching node -> recreate.
2. Since Kubernetes 1.17 the default scheduler places DaemonSet pods (older versions had the DaemonSet controller bind pods directly, bypassing the scheduler). The practical consequences are large: node affinity, taints/tolerations, and preemption all apply to DaemonSet pods just like any other pod. A DaemonSet pod needs tolerations to land on a tainted node; it can preempt lower-priority pods if it carries a high `PriorityClass`; the controller adds an implicit `nodeAffinity` for the specific node plus a few default tolerations (e.g. `node.kubernetes.io/not-ready`, `unreachable`, `disk-pressure`) so the agent stays put during node trouble.
3. `desiredNumberScheduled` counts only schedulable-for-this-pod nodes. That's why tainting a node without a matching toleration *reduces* `DESIRED` rather than producing a `Pending` pod. There is no pending pod to find - the node simply isn't counted. This is the mechanical reason the blind spot is silent.
4. Rolling updates respect `maxUnavailable`. With `updateStrategy: RollingUpdate` and `maxUnavailable: 1`, changing the pod template rolls one node at a time: delete the old pod on a node, wait for the new one to be Ready, move on. (The alternative, `OnDelete`, updates a node's pod only when you delete it by hand - useful for risky agents you want to roll manually.) There is no surge option the way Deployments have `maxSurge`, because you can't run two of the same DaemonSet pod on one node.
5. DaemonSets are how the cluster runs its own infrastructure. The CNI (aws-node / Cilium), `kube-proxy`, the CSI **node** plugins, and every log/metric agent are DaemonSets. The pattern isn't an app-deployment tool you happen to also use for agents - it *is* the platform layer.

The durable lesson: a Deployment's failure is loud (pods Pending, replicas short, events). A DaemonSet's failure is **quiet** - it faithfully covers the nodes it was told to, and says nothing about the nodes it wasn't. Coverage is the thing you must verify, and you verify it against the node list, never against the DaemonSet's own "healthy" status.

## Dev notes
- Node-local concerns do not belong inside your app pod. Log shipping, host metrics, packet capture, security agents - these are properties of the *node*, not your service. Putting a log forwarder as a sidecar in every app pod (lab 12) multiplies it by every replica and couples your release to the agent's. A DaemonSet runs it once per node, shared.
- The downward API gives the agent its identity. This lab injects `spec.nodeName` via `fieldRef` so each pod knows (and logs) which node it's on. Real agents do the same to tag metrics/logs with the node - that's how you later attribute a log line to a host.
- You almost never write a DaemonSet as an app developer - you consume the platform's (logs land in the aggregator, metrics in Prometheus). Knowing it exists explains *why* your stdout becomes searchable without you shipping anything.

## DevOps / Platform notes
- **The canonical agents:** `fluent-bit` or `vector` (log shipping), `node-exporter` (host metrics), `dcgm-exporter` (GPU metrics), the CNI and `kube-proxy`, CSI node plugins, Falco/security agents - all DaemonSets. `kubectl get ds -A` is a quick inventory of your platform layer.
- Resource requests multiply by node count. A DaemonSet pod runs on *every* node, so its `requests` are a per-node tax: `cpu: 100m` across a 200-node fleet is 20 cores reserved before a single app pod schedules. This lab pins tiny requests (`10m`/`16Mi`) for exactly this reason. Over-requesting an agent is a fleet-wide capacity leak.
- Rolling DS updates are your safe path to upgrade an agent. `maxUnavailable: 1` (or a percentage) bounds blast radius - a bad fluent-bit image takes down one node's logging at a time, and `kubectl rollout undo ds/...` reverts. Use `OnDelete` for agents where you want to canary node-by-node manually.
- **Targeting:** `nodeSelector` for the simple "only these pools" case; `nodeAffinity` when you need expressions (`In`, `NotIn`, `Exists`). On EKS, node groups carry `eks.amazonaws.com/nodegroup` labels; on OVH, label the node pool. Taint special pools and give the agent the matching toleration so coverage follows the hardware.

## Architect notes (trade-offs)
- **Two layers, two controllers.** The **platform layer (per-node infra) is DaemonSets; the application layer** (your services, your data) is Deployments and StatefulSets. Keeping them in different controllers, often different namespaces, with different RBAC and priority, is what lets you reason about "the cluster's own plumbing" separately from "what runs on it."
- DaemonSet vs. per-app sidecar (lab 12). A sidecar is a *per-pod* helper - it scales with replicas, sees only its pod, and ships with the app's lifecycle. A DaemonSet agent is a *per-node* shared service - one instance serves every pod on the node, decoupled from any app's release. Choose sidecar when the concern is pod-scoped (a per-request proxy, a config reloader); choose DaemonSet when the concern is node-scoped (host logs, host metrics, the node's network). The mesh debate (sidecar Envoy vs. eBPF/ambient per-node) is exactly this trade-off at scale.
- Coverage is a guarantee you design for. "Every node is observed" is only true if the agent matches and tolerates every node *by construction*. Architecting it means: agents tolerate all taints (or a documented, monitored exception), and a control catches a node with no agent. Don't leave it to whoever last edited a `nodeSelector`.

## SRE notes (failure modes, SLOs, toil)
- Node-level observability coverage is the whole point - and it fails silently. The two blind spots in this lab are the two real incidents: (A) a `nodeSelector` typo or pool rename so the agent matches nothing or fewer nodes; (B) a new tainted pool (GPU, spot, dedicated tenant) the agent doesn't tolerate. In both, `kubectl get ds` looks *healthy*; the gap only shows when you compare agent pods to the node list - usually at 3am, when the one node with the problem is also the one node you have no logs for.
- **The alert that catches it: `kube_daemonset_status_number_unavailable > 0` catches scheduling failures, but not** the matched-nothing case (there's nothing unavailable when nothing is desired). For that you need `kube_daemonset_status_desired_number_scheduled` compared against `kube_node_info` count - alert when a "should-be-everywhere" agent's desired count is below the schedulable node count. Treat coverage parity as an SLO.
- Agents must tolerate all taints, or you get blind nodes. The default course of action for an observability/security DaemonSet is to tolerate everything (`operator: Exists` with empty key, in the extreme) so coverage is unconditional. A deliberately-excluded node should be a documented, alerting exception - never an accident of a missing toleration.
- Footprint × node count is a capacity line item. When you size a node group, the DaemonSet tax (sum of every agent's requests) is reserved per node. A new heavyweight agent rolled fleet-wide can quietly push nodes into eviction pressure. Review agent requests as part of capacity planning, not as an afterthought.
- PriorityClass keeps the agent alive. Under node pressure the scheduler preempts low-priority pods first; a node agent at default priority can be evicted exactly when you need its telemetry most. This lab gives the tolerant agent a high `PriorityClass` so the node's eyes don't close during the incident.

## AI/ML notes (LLM/ML serving mapping - conceptual)
*(No GPU is used or required in this lab - these are the conceptual mappings.)*
- `nvidia-device-plugin` is a DaemonSet that runs on GPU nodes and advertises `nvidia.com/gpu` as an allocatable resource. Without it, the GPUs are physically present but **invisible to the scheduler** - pods requesting `nvidia.com/gpu` stay `Pending` forever. The device plugin is the bridge between hardware and the resource model, and it's a DaemonSet precisely because it's a per-node concern.
- **`dcgm-exporter` is a DaemonSet** that exports per-GPU telemetry (utilization, memory used, temperature, power, ECC errors) for Prometheus. This is the data behind GPU autoscaling decisions and serving SLOs (is the GPU the bottleneck, or is it TTFT/TPOT in the model server?).
- Both must tolerate the GPU node taint. GPU pools are typically tainted `nvidia.com/gpu:NoSchedule` (lab 14) so only GPU workloads land there. The device plugin and dcgm-exporter therefore *must* carry the matching toleration - the exact blind-spot mechanic from this lab. A device plugin that doesn't tolerate the GPU taint = a GPU pool that the scheduler thinks has no GPUs *and* no telemetry: the worst, most expensive blind node in the cluster.

## Pitfalls
- **DaemonSet missing tolerations** -> blind on tainted nodes (control-plane, GPU, spot, dedicated pools). The pod doesn't error - the node just leaves the denominator. Tolerate broadly for observability/security agents.
- **Forgetting `requests`** -> the agent's footprint × node count surprises capacity, and unbounded usage gets the agent (your telemetry) OOM-killed under pressure. Always set requests *and* a PriorityClass.
- **`nodeSelector` matching nothing** -> silent zero coverage; `get ds` looks healthy. Alert on desired-vs-node-count parity, not just on `unavailable`.
- **`hostPath` / `hostNetwork`** (common in real agents that read `/var/log` or the host network) are powerful and a security exposure: a compromised agent pod has node-level reach. Scope them tightly, use read-only mounts, and treat agent images as high-trust (lab 19).
- **Expecting `NoSchedule` to evict** -> it doesn't; only `NoExecute` evicts running pods. A newly-tainted node keeps its old agent until that pod restarts, so the blind spot can appear *later*, decoupled from the change that caused it.
