# Lecture 06 - Service Discovery: VIPs, EndpointSlices, CoreDNS & the readiness gate

## Answers to the lab checkpoints
- **(0)** **Three** - one per Ready backend pod. The Service selects `app.kubernetes.io/name=backend`, the Deployment runs 3 replicas, all pass readiness, so the EndpointSlice controller lists three pod IPs. Endpoints map to *Ready pods*, not to the Service or the replica count directly.
- **(1)** `nslookup backend` returns **exactly the `clusterIP`** you printed - they're the same. CoreDNS resolves the Service name to its VIP. The bare name `backend` worked because the client's `/etc/resolv.conf` has `search lab-06-services-dns.svc.cluster.local svc.cluster.local cluster.local ...`; the resolver appends each search domain until one resolves, so `backend` -> `backend.lab-06-services-dns.svc.cluster.local`. That's why same-namespace code can dial just `backend`, and cross-namespace code uses `backend.<ns>`.
- **(2)** You'll see **all 3** distinct pod names, in a **roughly random** (not strict round-robin) order. kube-proxy's iptables mode picks an endpoint with equal *probability* per new connection, so over 12 requests you get a random-looking spread that converges to even. It is **per-connection**, not per-request, balancing (more in *Architect* notes).
- **(3)** The EndpointSlice held **3** addresses at 3 replicas, **5** at 5 replicas, **3 again after scaling down - it tracks the count of Ready** pods at each moment. The `wget` loop at 5 replicas now surfaces up to 5 distinct names because there are 5 endpoints to DNAT to.
- **(4)** The dying pod's IP is **removed from the slice the moment it goes terminating/NotReady; the replacement's IP is added only after it passes readiness. The Service was not** down: 2 (or 4) Ready endpoints kept serving through the gap, so the `wget` loop saw no failures. Self-healing plus readiness gating means a single pod death is invisible to clients - exactly the property a Service exists to provide.
- **(5) `nslookup backend-headless` returns 3 `Address:` lines, and they are the pod IPs (match them to `get pods -o wide`), not** a VIP. `clusterIP: None` tells the cluster to allocate no VIP and instead publish one A record per Ready pod. The client now sees all backends and chooses one itself - there's no kube-proxy DNAT in the path.
- **(6) Every response could come from a pod on any node, not just the node you hit. By default a NodePort (and the ClusterIP underneath it) load-balances cluster-wide**: kube-proxy on the node you contacted DNATs to any Ready endpoint, hopping to another node if needed. (That extra hop is why some setups set `externalTrafficPolicy: Local` to keep traffic node-local and preserve the client source IP - at the cost of uneven balancing.)
- **(7)** **Yes to both.** `kubectl get svc backend-lb` shows a `clusterIP` *and* a `NodePort` in `PORT(S)` (e.g. `80:31234/TCP`) *and* an `EXTERNAL-IP`. A `LoadBalancer` Service is built **on top of** a NodePort which is built on top of a ClusterIP - three layers. The cloud LB forwards external traffic to the node ports; kube-proxy takes it from there. That's why deleting the Service (not just the namespace) is what releases the cloud LB.
- **(B1)** **Zero addresses - the EndpointSlice drains empty because no pod is Ready. The client's `wget` fails with a connection refused / timeout (`FAILED: 1`): DNS still resolves the VIP, but kube-proxy has no endpoint to DNAT to, so packets to the VIP are rejected. Crucially, `kubectl get/describe svc backend` reports nothing wrong - the Service object is healthy; it just has no backends. The signal lives in the EndpointSlice**, not the Service.
- **(B2) DNS resolves `backend-typo` to its VIP, the `wget` fails the same way (no endpoints to route to), and nowhere** does Kubernetes raise an error - no event, no bad status. `describe svc backend-typo` shows `Selector: app=backend` and `Endpoints: <none>`, while `backend` shows its 3 endpoints. The only tell is the empty `Endpoints`/EndpointSlice. This is the most common self-inflicted "it's down but everything looks green" incident.
- **(B3) Aligning the selector repopulated the EndpointSlice instantly - no restart, because endpoint membership is recomputed from labels continuously, exactly like any label selector in lab 01. The first command** for either failure is `kubectl get endpointslices -l kubernetes.io/service-name=<svc> -o wide` (or `kubectl get endpoints <svc>`): empty endpoints localizes the fault to *backends*, then you check (a) do pods exist? (b) are they Ready? (c) does the selector match their labels? - three checks, in that order.

## What just happened (under the hood)
A Service is not a process and not a proxy box - it's a record in etcd plus rules programmed into the data plane. Three moving parts make `wget backend` reach a pod:

1. **The Service VIP is virtual. When you create a `ClusterIP` Service, the API server allocates an IP from the service CIDR. Nothing listens on it.** No pod, no node, no process owns that address. It exists only as a target for DNAT rules.
2. **The EndpointSlice controller watches pods matching the Service's `selector` and continuously writes the set of Ready pod IPs into one or more `EndpointSlice` objects (labelled `kubernetes.io/service-name=<svc>`). A pod enters the slice when its readiness probe passes and leaves the instant it goes NotReady, terminating, or stops matching the selector. This controller is the whole "is this pod in the Service?" decision** - and it keys entirely off readiness + label match.
3. **kube-proxy runs on every node, watches Services and EndpointSlices, and programs the kernel: in iptables mode it writes DNAT rules that, for a packet destined to the VIP, rewrite the destination to one randomly chosen endpoint IP (equal probability); in IPVS mode it uses an in-kernel load-balancer table (real scheduling algorithms, better at thousands of services). Either way the rewrite is connection-level (L4)**: the endpoint is chosen once, when the connection opens, and every packet of that connection goes to the same pod.

So `wget backend/hostname` does: CoreDNS resolves `backend` -> VIP; the client opens a TCP connection to the VIP:80; the kernel's kube-proxy rule DNATs that to `<some-ready-pod>:8080`; the pod answers with its own hostname. Do it again and a new connection may be DNAT'd to a different pod - that's the "load balancing" you watched.

Headless removes the VIP and the DNAT entirely. `clusterIP: None` -> no VIP allocated -> kube-proxy programs nothing -> CoreDNS returns the pod IPs directly. The *client* now holds the list and picks. This is the right model when callers need to address **individual** pods (StatefulSet peers, lab 10) rather than "any healthy backend."

Readiness is the seam that connects lab 02/03 to this lab. In lab 02 a `Running`-but-NotReady pod stalled a rollout; here the *same* gate keeps that pod out of the Service's endpoints. One mechanism, two consequences: readiness controls **rollout progress** and **traffic eligibility**. A pod that lies about readiness (probe returns 200 before it can serve) gets traffic it can't handle; a pod with no probe at all is "Ready" the instant it's Running and joins endpoints immediately - often before it can serve a request.

**Why EndpointSlice, not Endpoints?** The legacy `Endpoints` object packed *every* backend of a Service into a single object. At thousands of endpoints, any pod change rewrote the whole object and every kube-proxy/watcher in the cluster re-downloaded it - an O(N) thundering herd on each scale event. `EndpointSlice` shards endpoints into chunks (≤100 by default), so a single pod change touches one small slice. `Endpoints` still exists for backward compatibility; EndpointSlice is the source of truth at scale.

## Dev notes
- **Depend on DNS names, never pod IPs.** Pod IPs are ephemeral - they change on every restart, reschedule, and scale event. The stable contract is `http://<svc>` (same namespace) or `http://<svc>.<ns>` (cross-namespace), or the FQDN `<svc>.<ns>.svc.cluster.local`. Hardcoding an IP works in the demo and pages you at 3am.
- **Know which DNS form to use.** Same namespace: bare `backend`. Different namespace: `backend.other-ns`. Don't rely on the search-domain magic across namespaces - be explicit. CronJobs/Jobs in other namespaces especially trip on this.
- L4 vs L7 - what a Service can and cannot do. A Service balances **connections, not requests. A client that opens one long-lived connection (HTTP/2, gRPC, a pooled DB connection) is pinned to one** pod for that connection's life - so naive gRPC clients hit a single backend while others idle. Per-*request* balancing, path/host routing, retries, and header-based routing need **L7** (Ingress in lab 07, or a service mesh). If you find yourself wanting "route `/v2` to the new pods," you've outgrown a Service.

## DevOps / Platform notes
- One cloud LB per `LoadBalancer` Service = one bill per Service. Twenty `type=LoadBalancer` Services = twenty NLBs/OVH LBs. The standard pattern is **one** LB fronting an **Ingress controller** (lab 07) that fans out to many ClusterIP Services over L7 - far cheaper and gives you host/path routing for free. Reserve raw `type=LoadBalancer` for non-HTTP (TCP/UDP) or when you genuinely need a dedicated L4 entry point.
- The LB->Service lifecycle is the cloud-controller's job, and it leaks. The cloud-controller-manager creates the LB when you create the Service and deletes it when you delete the Service. If you delete the *namespace* (or the cluster) while the controller is unhealthy or already gone, the LB can be **orphaned and keep billing**. Always delete `type=LoadBalancer` Services *before* tearing down - which is exactly the Cleanup ordering in this lab and in `eks.md`/`ovh.md`.
- CoreDNS is a critical, shared dependency. It runs as a Deployment in `kube-system`; if it's down or throttled, *every* name lookup in the cluster fails and the symptom looks like "everything is broken at once." Give it requests/limits, run ≥2 replicas, watch its latency, and know that `ndots:5` in the default `resolv.conf` makes each external lookup try several search domains first (a real latency/QPS cost - tune with `dnsConfig`/`ndots` for chatty external callers).
- **EndpointSlice vs Endpoints at scale** is an operational knob, not just trivia: a Service with thousands of backends behaves very differently on control-plane load depending on which the watchers consume. Modern kube-proxy uses slices; old tooling that reads `Endpoints` can become a scaling bottleneck.

## Architect notes (trade-offs)
- **East-west vs north-south.** `ClusterIP` is **east-west - service-to-service inside the cluster, the overwhelming majority of traffic, free and invisible. `NodePort`/`LoadBalancer`/Ingress are north-south** - traffic crossing the cluster boundary. Decide which a Service is *for*; don't expose an internal microservice with a public LB because "it was easy."
- **VIP (ClusterIP) vs headless.** Use a **VIP when callers want "any healthy backend" and you want kube-proxy to spread load and hide individual pods - the default for stateless services. Use headless** when callers must address **specific, stable** pods: StatefulSet members (`pod-0`, `pod-1`, ..., lab 10), sharded stores, peer-to-peer clusters (Cassandra, Kafka), or any client doing its own client-side load balancing over the full endpoint set (common for gRPC). Headless trades kube-proxy's L4 balancing for client control and per-pod addressability.
- `externalTrafficPolicy: Cluster` vs `Local`. `Cluster` (default) balances evenly but adds a possible extra node hop and hides the client source IP (SNAT). `Local` keeps traffic on the node that received it (preserves source IP, lower latency) but balances unevenly if pods aren't spread across all nodes, and drops traffic to nodes with no local pod. Pick per Service based on whether you need the real client IP.

## SRE notes (failure modes, SLOs, toil)
- "The service is down but the pods are up" -> check endpoints FIRST. This lab's whole point. The Service object can be green while clients get connection-refused. Your reflex:
 1. `kubectl get endpointslices -l kubernetes.io/service-name=<svc> -o wide` - empty? The fault is *backends*, not the Service.
 2. Empty because pods are **NotReady**? (`kubectl get pods` shows `0/1`) -> it's a readiness/probe/dependency problem (Break-it A). Same root cause as a stalled rollout in lab 02.
 3. Empty because the **selector matches nothing**? (`kubectl describe svc` -> `Selector:` vs pod labels) -> label/selector typo (Break-it B). No restart needed; fix the selector.
 This three-way split (no pods / not Ready / no match) resolves the majority of "service unreachable" pages in minutes.
- A readiness probe is a traffic gate, so design it as one. Too lax -> you admit unready pods to endpoints and serve errors. Too strict / flapping -> pods bounce in and out of the slice, causing intermittent failures and connection churn that's miserable to debug. Probe the real serving path, with sane `failureThreshold`/`periodSeconds`.
- CoreDNS outage = total, cluster-wide resolution failure. It presents as "everything is failing" with no single culprit. Alert on CoreDNS availability and latency specifically; a DNS brownout is one of the highest-blast-radius failures in a cluster.
- LB health-check mismatch is a sneaky north-south failure. A cloud LB runs its **own** health checks against the node ports, independent of Kubernetes readiness. If the LB's health-check path/port is misconfigured, it can mark healthy nodes down (no traffic flows despite Ready pods) or unhealthy nodes up (traffic to nowhere). When external traffic fails but in-cluster `wget` works, suspect the LB health check, not the pods.
- **Toil:** manually mapping which Service fronts which pods across namespaces is toil; lean on labels and `kubectl get endpointslices`/service-topology tooling rather than spelunking by hand.

## AI/ML notes (LLM/ML serving mapping - conceptual)
- Model serving discovers replicas exactly like this. A fleet of identical model-server replicas (vLLM, TGI, Triton) behind a `ClusterIP` Service is discovered by DNS name, and kube-proxy L4-balances **connections** across the Ready replicas - the same mechanism you watched with `/hostname`. A replica that's still loading weights / building KV-cache stays NotReady and out of endpoints, so no request hits a cold replica and eats a multi-second first-token latency (the readiness gate from lab 02/03, applied to serving).
- Headless Services for replica-addressable / sharded inference. When inference is **sharded (tensor/pipeline-parallel across pods, or a KV-cache routed to a specific shard) each shard must be reachable by a stable, individual** name - that's a headless Service over a StatefulSet (lab 10): `shard-0.model.ns.svc.cluster.local`, `shard-1...`. A router then sends each request to the shard that holds the relevant state, rather than "any backend." VIP balancing would defeat the point.
- **An L4 Service in front of model replicas is a *connection* balancer - usually not enough. Because Services balance per-connection, a router that keeps long-lived HTTP/2 connections to backends will pin to one replica; and you typically need L7** to route by model name, by path (`/v1/chat`), by token budget, or to do prompt/semantic caching and request-level retries. That's an Ingress / gateway / mesh (lab 07), not a Service. Use the Service for "find the healthy replicas," and L7 for "route this *request* intelligently." (All conceptual - no GPU is required anywhere in this course.)

## Pitfalls
- Selector/label mismatch -> silently empty endpoints. No error, no event - just a Service that resolves but never answers. Always diff `describe svc`'s `Selector:` against the pods' actual labels (Break-it B).
- Assuming the Service does per-request load balancing. It's **per-connection** (L4). Long-lived HTTP/2/gRPC/DB connections pin to one pod. For per-request spread you need L7 or client-side balancing over a headless Service.
- **Hardcoding pod IPs** (or even a `clusterIP`, which is stable but not portable across clusters/namespaces). Use the DNS name; it's the only durable contract.
- **Forgetting `type=LoadBalancer` cost** - one billed LB per Service, and an orphaned LB if you delete the namespace before the Service. Front many services with one Ingress LB instead (lab 07).
- **No readiness probe -> pods join endpoints the instant they're Running and take traffic before they can serve. A flapping probe** -> pods churn in/out of the slice causing intermittent failures.
- Confusing readiness with liveness here too: liveness restarts a sick pod; readiness pulls it out of endpoints (no restart). Break-it A used readiness on purpose so pods stayed `Running` while leaving the Service.
