# Lecture 07 - Ingress, TLS at the edge & the caching tier as a first-class component

## Answers to the lab checkpoints
- **(0)** **One cloud LB total, and you didn't create it in this lab - it's the `ingress-nginx-controller` `Service` (`type=LoadBalancer`) from cluster-setup. North-south traffic enters there: the cloud LB (L4) hands packets to the controller, and the controller** does all the L7 routing for *every* Ingress in the cluster. Adding Ingress objects adds **routing rules**, not load balancers. That's the whole economic point of Ingress: one billed front door, N routes.
- **(1) `EXTERNAL-IP` is the cloud LB fronting the controller - an NLB hostname** on EKS, an **IP** on OVH. Ports `80` and `443` both terminate on the controller. No app has its own external address; they sit behind ClusterIP Services and are only reachable *through* the controller (or in-cluster).
- **(2)** Each backend is a Deployment + **ClusterIP** Service - cluster-internal only, **no `EXTERNAL-IP`**. The Service gives a stable virtual IP/DNS name and L4-load-balances across the 2 Ready pods; exposure to the outside is entirely the Ingress controller's job.
- **(3)** A request to `http://$LB/a` with **no `Host` header (or the wrong host) does not match the rule - the rule is scoped to `host: lab07.demo.local`. ingress-nginx matches host first, then path; a request whose Host doesn't match any rule falls through to the controller's default backend -> 404**. That's why we use `curl --resolve` (which sets both the connection target *and* the `Host` header) rather than curling the raw LB IP.
- **(4)** `/a/version` -> podinfo **6.7.0** (blue); `/b/version` -> **6.7.1 (green). Identical host, LB, and port - only the path prefix chose the backend, and that choice was made by nginx inside the controller at L7** (it can read the URL path; an L4 LB cannot). The `rewrite-target: /$2` stripped the `/a` prefix so podinfo saw `/version`, not `/a/version`.
- **(5)** cert-manager produced a **Secret** named `lab07-tls` of type **`kubernetes.io/tls` (holding `tls.crt` + `tls.key`). You declared a desired certificate; the controller reconciled it by asking the `Issuer` to sign and writing the Secret - the same declarative loop as everything else, applied to PKI. No `openssl`, and it will auto-renew** before `renewBefore`.
- **(6) `https://.../a/version` returns 6.7.0 over TLS; the served cert's `subject`/`issuer` is the self-signed `lab07.demo.local` cert from the Secret; and plain HTTP returns 308 redirecting to the `https://` URL (ingress-nginx's default `ssl-redirect` once a `tls:` block exists). TLS is terminated at the controller** - the hop from controller to ClusterIP backend is cleartext inside the cluster.
- **(7)** First request `X-Cache-Status: MISS` (nginx had nothing cached, so it called podinfo and paid the 100ms `/delay`), second `HIT` (served from nginx's cache zone, no backend call, no delay). Same URL -> same cache key -> HIT. The header is your X-ray into what the cache did.
- **(8)** Of ~1600 requests, **a single-digit handful** reach backend-a - ideally **one** (the first MISS), plus a few when the 60s `proxy_cache_valid` window rolls over or `proxy_cache_lock` lets one revalidation through. ~99.7% were served from nginx's cache. That ratio *is* the offload: the backend saw orders of magnitude fewer requests than the client sent.
- **(B1)** With each request sleeping 100ms, one replica can serve ~10 req/s *per concurrent slot*. At 80 req/s across 2 replicas you need ~8 requests in flight at once just to break even - and podinfo's request handling plus connection limits don't give you that headroom cheaply. So requests **queue**, **p99 latency climbs** toward (and past) the 2s fortio `-timeout`, and you start seeing `Code -1` (timeouts) and non-200s. The cached run had none of this because nginx absorbed the load.
- **(B2) Cached run: p99 stays low (HITs are microseconds) and ~0 timeouts. Direct run: the `99%` percentile spikes (often to the timeout ceiling) and `Code -1`/non-200 counts appear. The backend's per-request work didn't change - it ran out of concurrency/throughput**. Latency is a *queueing* phenomenon: arrival rate exceeded service rate, so the queue (and tail latency) grew.
- **(B3)** The backend pods stayed **Running** and **Ready** - they were **saturated, not crashed**. `kubectl top` shows CPU pushing toward the limit; no restarts, no `CrashLoopBackOff`. Saturation is a capacity failure, not a health failure - which is exactly why liveness/readiness probes (lab 02/03) don't save you here, and why the cache (or autoscaling, lab 18) is the real fix.

## What just happened (under the hood)
There are **two** different "ingress nginxes" and conflating them causes most Ingress confusion:

1. **The Ingress object** is just **configuration** - a declarative list of host/path -> service rules plus an optional `tls:` block, stored in etcd. By itself it does nothing; it routes no packets.
2. **The ingress-nginx controller** is a Deployment of real **nginx** processes that **watches Ingress objects (filtered by `ingressClassName: nginx` <-> the `IngressClass` it owns) and rewrites its own nginx config** every time one changes. *That* nginx is what actually terminates connections and proxies traffic.

So the data path for `https://lab07.demo.local/a/version` is:

```
client ──TLS──▶ cloud LB (L4: just forwards :443) ──▶ ingress-nginx controller
                                                         │  terminates TLS (lab07-tls Secret)
                                                         │  matches host lab07.demo.local
                                                         │  matches path /a, rewrites to /version
                                                         ▼
                                              Service backend-a (ClusterIP, L4) ──▶ one of 2 podinfo pods
```

The cloud LB is **L4** - it knows IPs and ports, not URLs or hostnames (well, SNI aside). All the *interesting* routing - host, path, TLS, headers, rewrites - happens at **L7 in the controller. Below the controller, the ClusterIP Service** is L4 again, doing simple round-robin (via kube-proxy/iptables/IPVS or eBPF) across the pods in its EndpointSlice (lab 06). Edge does L7; Services do L4; the layers compose.

**cert-manager** closes the PKI loop declaratively. You wrote an `Issuer` (here a self-signed CA - in prod a `ClusterIssuer` backed by Let's Encrypt ACME or your org CA) and a `Certificate` (the *request*: this name, these DNS SANs, store it in this Secret). cert-manager's controller signs it, writes a `kubernetes.io/tls` Secret, and - the day-2 payoff - **renews it before expiry** and re-writes the Secret in place; the controller hot-reloads. Certificate rotation stops being a calendar reminder and becomes a reconcile loop.

**The caching tier** is the middle of an **edge -> cache -> app topology. nginx with `proxy_cache` stores upstream 200s keyed by `$scheme$request_method$host$request_uri` for 60s. A repeat request with the same key is a HIT - served from nginx, the backend never sees it. `proxy_cache_lock on` is the subtle, important knob: when the cache is cold (or expires) and 100 requests miss simultaneously, lock collapses them into one backend call and makes the other 99 wait for it - the built-in defense against a cache stampede**.

**Gateway API** is the successor to Ingress. Ingress is one overloaded object with per-vendor annotations doing all the work (you used four `nginx.ingress.kubernetes.io/*` annotations above for things the spec can't express). Gateway API splits this into role-oriented, typed resources (`GatewayClass`, `Gateway`, `HTTPRoute`, `TLSRoute`) - header matching, traffic splitting, and cross-namespace delegation become first-class fields instead of annotations. New ingress features land there now; Ingress is in maintenance.

## Dev notes
- **Assume you're behind a proxy. The client IP your app sees is the controller's, not the user's - read `X-Forwarded-For` / `X-Forwarded-Proto`** for the real client and original scheme (the controller sets them). Logging `RemoteAddr` directly logs the proxy.
- **Make handlers idempotent. Behind an LB + edge proxy, requests get retried** (connection resets during a rollout, proxy timeouts). A `GET`/`PUT`/`DELETE` is naturally safe to retry; a `POST` that charges a card is not - use idempotency keys.
- Get cache-control right or your cache is a liability. The cache keys on the URL; if `/a/data?token=...` varies per user but you cache it, you'll serve one user's response to another. Set `Cache-Control: private`/`no-store` on personalized responses, sensible `max-age` on shared ones, and include any varying dimension in the cache key (or `Vary:`). A cache is only as correct as the cache-control contract the app honors.
- The path rewrite is real and visible. `rewrite-target: /$2` means podinfo received `/version`, not `/a/version`. If your app generates absolute links or sets cookies scoped to a path, the prefix-stripping bites you - prefer host-based routing (`a.example.com`, `b.example.com`) over path-prefix routing for apps that aren't path-agnostic.

## DevOps / Platform notes
- **One controller, many Ingresses.** The platform team runs ingress-nginx once; every app team submits an Ingress with `ingressClassName: nginx` and gets routing on the shared front door. This is the cost/operability win - and a blast-radius concern: a bad regex annotation or a huge config can disrupt *everyone's* routing through that controller.
- Cert rotation is a reconcile loop, not a runbook. cert-manager renews before `renewBefore` and rewrites the Secret; the controller reloads. Monitor `certmanager_certificate_expiration_timestamp_seconds` and alert on Certificates that are `Ready=False` or near expiry. The failure you're preventing is the classic one in the SRE notes.
- ingress-nginx vs cloud ALB Ingress (`### EKS`). The **AWS Load Balancer Controller satisfies an `Ingress`/`ingressClassName: alb` by provisioning a real ALB - L7 routing at the cloud edge, no in-cluster nginx hop, integrates with WAF/Shield/ACM-managed certs. Trade-off: it's not portable (ALB-specific annotations, one ALB per Ingress unless you use `IngressGroup`) and TLS/caching/rewrites behave differently. We chose ingress-nginx** so the manifests are byte-identical on OVH (`### OVH`: same chart, OVH LB underneath). Pick per environment: ALB for AWS-native shops that want managed edge; ingress-nginx for portability and a uniform config language.
- **Plan the Gateway API migration.** New features (advanced traffic splitting, header-based routing) are landing in Gateway API. ingress-nginx and most clouds ship a Gateway controller; greenfield routing should evaluate `HTTPRoute` over Ingress annotations.

## Architect notes (trade-offs)
- The caching tier is a component, not an afterthought. You designed `edge -> cache -> app` on purpose: the cache is where you decide *what's cacheable, for how long, keyed by what*. Bolting a cache on later means retrofitting cache-control semantics across every endpoint. Decide cacheability at design time, per route.
- **Where to terminate TLS.** Three choices: (a) **at the edge (what we did) - simplest, one place to manage certs, cleartext inside the trusted cluster network; (b) end-to-end / re-encrypt - edge terminates then re-encrypts to the backend, for compliance regimes that forbid any cleartext; (c) mTLS everywhere** via a service mesh (Istio/Linkerd, lab 08's "where to go next") - every hop authenticated and encrypted, at real CPU and operational cost. Edge-termination is the right default; push deeper only when a threat model or regulation demands it.
- L7 edge vs L4 Service is a deliberate layering. Keep smart, app-aware routing at *one* L7 tier (the controller) and let everything below be dumb, fast L4. Spreading L7 logic across many tiers (edge does host routing, cache does path routing, app does header routing) makes traffic impossible to reason about.
- Replicated Load-Balanced Service (the pattern). Stateless, identical replicas behind a load balancer; the LB spreads load and the replica count is your horizontal scale and redundancy. The cache is an optimization layer *on top of* this pattern - it doesn't replace replication, it reduces the load replication has to absorb.

## SRE notes (failure modes, SLOs, toil)
- Cert expiry is the classic 3am page. An expired edge cert takes down **every** HTTPS route at once, and it happens at a *predictable* time you'll have forgotten. cert-manager auto-renewal removes the toil, but you still **monitor expiry** (a stuck ACME challenge or a `Ready=False` Certificate means renewal silently failed). Alert on days-to-expiry, not on the outage.
- Tail latency is dominated at the edge. Under load (Break-it), p99 exploded not because requests got slower but because they **queued - arrival rate exceeded service rate. The edge is where you see and control this: edge rate-limiting** (`limit_req`, ingress-nginx `limit-rps` annotations) sheds load *before* it cascades into a saturated backend that then drags down everything sharing those pods. A small amount of 429-at-the-edge prevents a total downstream collapse.
- **Cache stampede / thundering herd.** When many cache entries expire at the *same instant* (synchronized TTLs) or a popular key expires under high concurrency, every miss hits the backend simultaneously and can knock it over - the exact saturation from Break-it, triggered by a cache *expiry* instead of cache *removal*. Defenses: `proxy_cache_lock` (request collapsing - you enabled it), **stale-while-revalidate (serve stale while one request refreshes), and TTL jitter** so entries don't expire in lockstep.
- **Saturation ≠ unhealthy.** The backend stayed Ready throughout Break-it. Probes catch *crashes*, not *overload*. Your SLI for this is **latency + error rate + utilization**, and the fixes are capacity (HPA, lab 18), shedding (edge rate-limit), or offload (the cache) - not a liveness probe.

## AI/ML notes (LLM/ML serving mapping - conceptual)
- Response caching in front of an LLM is the single biggest cheap win. Identical prompts (and there are *many* in production - boilerplate system prompts, repeated FAQ-style queries) can be served from a cache instead of re-running a multi-second, dollar-costing generation. The HIT/MISS you watched is exactly this: a HIT means **zero TTFT and zero token cost**. **Semantic caching** generalizes it - embed the prompt and serve a cached response for *near-duplicate* prompts (cosine similarity above a threshold), not just byte-identical ones.
- An LLM gateway sits exactly where this Ingress+cache sits. Tools like an LLM gateway/router do, at the same edge position: **model-version routing (route to `gpt-x` vs `llama-y` by header/path - the same host/path routing you did), token-based rate limiting (rate-limit by tokens, not requests, because cost is per-token), response/semantic caching**, and TLS/auth termination. Conceptually it's this lab's controller + cache, specialized for token economics.
- Two different caches - don't confuse them. This lab's cache is the **outer response cache (HTTP responses, keyed by request). Inside the model server live KV-cache (per-request attention state during one generation) and prefix/prompt-cache** (reuse of a shared prompt prefix's KV across requests, e.g. vLLM automatic prefix caching). The response cache eliminates the *whole* call; the prefix/KV caches speed up calls that *do* run. They stack: response cache first, prefix cache behind it. (All conceptual - **no GPU needed for this lab.**)
- **Stampede is worse for LLMs.** A cache miss here costs 100ms; a miss on an LLM gateway costs seconds and real money, and a synchronized expiry can saturate scarce, expensive GPU replicas instantly. Request collapsing + stale-while-revalidate matter *more* in front of model servers, not less.

## Pitfalls
- Wrong `ingressClassName` -> silent no-op. If the Ingress's class doesn't match the controller's `IngressClass`, the controller **ignores it** - no routing, no error, just 404s. The most common "my Ingress does nothing" cause. Confirm with `kubectl get ingressclass` and the controller's `--ingress-class` flag.
- Curling the LB IP without the `Host` header -> 404. Host-scoped rules need the right `Host`; `--resolve` sets it for you, a raw `curl http://<ip>/a` does not (checkpoint 3).
- **Terminating TLS *everywhere* (mesh + re-encrypt on every hop) for no threat-model reason -> CPU burn and operational complexity. The opposite - terminating nowhere** (cleartext to the internet) - is a breach. Match termination to your actual trust boundary; edge is usually right.
- Cache stampede on synchronized expiry -> a fleet of entries expiring together turns your cache from a shield into a load amplifier. Use request collapsing, stale-while-revalidate, and TTL jitter.
- Assuming session affinity across tiers. ingress-nginx can pin a client to a backend pod (cookie affinity), but a cache tier in between breaks the assumption (the cache serves everyone the same bytes), and any app that *requires* sticky sessions to be correct is hiding state that belongs in a shared store. Affinity is a performance hint, not a correctness guarantee - design stateless and put session state in Redis/DB.
- Path-prefix routing + apps that emit absolute paths -> broken links/cookies after `rewrite-target` strips the prefix. Prefer host-based routing for non-path-agnostic apps.
