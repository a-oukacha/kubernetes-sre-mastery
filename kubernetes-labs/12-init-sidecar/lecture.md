- **(4)** The content **changes over time - the `refresh` sidecar rewrites `index.html` every 5 s. The pod is `READY 2/2`: every entry in `containers[]` (app and** classic sidecar) counts toward readiness.

## What just happened (under the hood)
A pod is not one container - it is a **shared execution context**: one network namespace (so every container reaches the others over `localhost`), optionally shared volumes (the `emptyDir` here), and a single lifecycle the kubelet drives. Lab 12 exercised the three ways to put more than one container in that context.

Init containers are a sequential, must-succeed prologue. The kubelet runs `initContainers[]` **one at a time, in array order. Each must exit 0 before the next is created; the regular `containers[]` are not created until every** init container has succeeded. That is why `init-app` showed `Init:0/1` with no app container in sight, and why the broken pod's `.status.containerStatuses` was empty - the kubelet literally never reached the app stage. An init container that fails is retried per the pod's `restartPolicy` (for a bare pod, `Always` -> it backs off and loops as `Init:CrashLoopBackOff`), and the app stays unborn until it succeeds. This is the ordering *guarantee*: setup either completes or the pod doesn't run a half-configured app.

Classic sidecars are concurrent peers with no ordering. A classic sidecar is just an extra entry in `containers[]`. The kubelet starts all `containers[]` together once init is done; there is **no guarantee the sidecar is up before the app, and no** guarantee it outlives the app on shutdown. All `containers[]` count toward pod readiness (`2/2`), and each restarts independently - which is why killing the app restarted only the app while `refresh` sailed on. The historical pain: a logging or proxy sidecar that *should* be up before the app (so early requests are proxied) or *should* drain after the app (so last logs ship, last requests flush) had no way to express that. On shutdown the kubelet sent SIGTERM to all containers at once; a proxy could die before the app finished draining. On a **Job**, a never-exiting classic sidecar even kept the Job from ever completing.

Native sidecars fix the lifecycle (GA 1.29). The fix is delightfully small: put the sidecar in `initContainers[]` and give it `restartPolicy: Always`. That one field changes the semantics - Kubernetes runs it in init order (so it starts **before** the app), but does **not** block on it exiting (because it never does); it lets the app start as soon as the sidecar has *started*. The sidecar then runs for the whole pod, restarts independently if it crashes, and is terminated **after the app containers on shutdown (drain-after). Because it lives in `initContainers[]`, it is not** counted in the ready-container total - that is why `native-sidecar` was `1/1`, not `2/2`. The pattern you've been told to use for mesh proxies, log shippers, and config-syncers finally behaves the way those use cases always needed.

Two durable lessons:
- **Pick the slot for the *timing* you need, not the function.** Setup that must finish first -> init container. Helper that must outlive/predate the app -> native sidecar. Helper that just needs to run alongside and has no ordering requirement -> classic sidecar is fine (and simplest).
- Containers in a pod are co-scheduled and co-located but independently restarted. Shared network and volumes; separate restart lifecycles. That combination is the whole composition toolkit.

## Dev notes
- Separation of concerns: don't bake helpers into the app image. Log shipping, config fetching, weight downloading, TLS termination - these are *operational* concerns. Keeping them out of the app image keeps the image small, the app single-purpose, and the helper independently versioned and reusable across many apps.
- Use init for setup, sidecars for ongoing help. Init container: "fetch config / migrate schema / wait for a dependency / download an artifact, then get out of the way." Sidecar: "keep doing X for as long as the app runs." If your "init" needs to keep running, it's a sidecar, not an init.
- **The localhost contract.** Inside a pod, containers talk over `localhost` and share volumes. Your app can assume a sidecar at `localhost:<port>` and a shared path - that's the API between them. Keep it explicit (document the port and path) so the helper stays swappable.

## DevOps / Platform notes
- **Service-mesh sidecar *injection* is this pattern, automated. Istio and Linkerd add an Envoy/proxy sidecar to your pods via a mutating admission webhook** - you label a namespace, and every pod silently gains a sidecar `containers[]` entry. You didn't write it; the platform injected it. Knowing this pattern is knowing what the mesh is doing to your pod spec.
- Native sidecars finally make mesh and logging behave on shutdown. Pre-1.29, the injected proxy could die before the app drained (dropped requests during termination) and on Jobs the proxy kept the Job "running" forever. Meshes have moved to native sidecars; if you operate one, confirm your clusters are ≥1.29 and the mesh is configured to use native sidecars.
- **Sidecar sprawl is a real cost.** Every pod that gets a proxy + a log shipper + a metrics adapter now runs three extra containers - three sets of requests, three restart surfaces, three images to patch. At fleet scale that is a meaningful slice of cluster spend and CVE surface. Audit what you inject and where.

## Architect notes (trade-offs)
- **Composition over modification.** Rather than fork the app to add a feature, attach a generic helper container: a `git-sync` sidecar that pulls config into a shared volume, a log-tailer, an OAuth proxy. One reusable helper image serves many apps - the structural patterns (init, sidecar, adapter, ambassador) are all "compose, don't modify."
- When a sidecar should be a DaemonSet instead. A per-*pod* sidecar pays its tax on every replica. If the concern is genuinely **per-node (node log collection, node metrics, a CNI/CSI agent), a DaemonSet** (lab 15) runs one copy per node instead of one per pod - far cheaper at scale. Rule of thumb: per-pod state/identity -> sidecar; node-level concern -> DaemonSet.
- **Init vs sidecar vs Job. A one-shot cluster-wide migration is a Job**, not an init container (an init runs on *every* pod start, including every restart and every replica). An init container is for per-pod setup that must precede *this* pod's app.

## SRE notes (failure modes, SLOs, toil)
- The sidecar resource and latency tax. Every pod now pays for the sidecar's CPU/memory requests, and every request may traverse an extra proxy hop (added p50/p99 latency). Multiply by replica count. Budget sidecar requests deliberately - and remember a sidecar **with no requests** can make a pod unschedulable or get the whole pod OOM-killed when the sidecar balloons.
- Classic sidecar crash semantics page you in surprising ways. Because classic sidecars count toward readiness, a crash-looping log shipper can flap the *whole pod* out of the Service even though the app is perfectly healthy - your app "goes down" because a non-critical helper died. Conversely, on shutdown a classic sidecar may die before the app drains. Native sidecars remove both surprises (not counted in readiness; drain after the app), which is the main operational reason to migrate.
- **The pre-1.29 outage shape.** Mesh adopters hit it repeatedly: the proxy sidecar started *after* the app, so the app's first outbound calls failed (no proxy yet); or the proxy was SIGTERMed alongside the app, so in-flight requests dropped during rollouts. If you still run classic mesh sidecars, those are your 3am failure modes.
- Init that downloads on every restart amplifies load. An init container that pulls a large artifact (a model, a dataset, a big config) runs again on *every* pod restart and *every* new replica. A crash loop or an autoscale event becomes a thundering herd against your artifact store. Cache it (a PVC, a node-local cache, or a CDN), or you've built a self-inflicted DDoS.

## AI/ML notes (LLM/ML serving mapping - conceptual)
- **Model-puller init container. The canonical use: an init container downloads model weights from object storage into a shared `emptyDir` (or, better, a PVC - see the toil note above) before** the inference server (vLLM, TGI, Triton) starts. The server image stays small and generic; weights are versioned and swapped independently of the server image. Exactly the `seed` -> `app` handoff you ran, scaled to gigabytes.
- Telemetry/metrics sidecar beside an inference server. A metrics-exporter sidecar (DCGM exporter for GPU stats, or an adapter normalizing vLLM's metrics to Prometheus format - lab 13) sits next to the model server, sharing `localhost`, exposing `/metrics` for the scrape. The model server doesn't need to know about your monitoring stack.
- **Native sidecar for a serving proxy. A request-routing or auth proxy in front of the model server must be up before the server accepts traffic and must drain after** it so in-flight generations finish - precisely the native-sidecar lifecycle. For long-running token generation, draining-after matters even more than for typical web requests: you don't want to SIGKILL a half-finished completion.

## Pitfalls
- A classic sidecar that must outlive (or predate) the app -> use a **native sidecar** (`initContainers` + `restartPolicy: Always`). Classic sidecars give you no ordering; don't fight it, switch slots.
- An init container that re-downloads large data on every restart -> cache it on a PVC or node-local store. Every restart and every new replica re-runs the init.
- **Assuming sidecars start in order** -> only init containers and **native** sidecars have start ordering. Classic `containers[]` start concurrently; if your app calls the sidecar at startup, a classic sidecar may not be ready yet.
- Forgetting the sidecar's resource requests -> the pod won't schedule (or will get OOM-killed). The sidecar's requests are part of the pod's total footprint.
- Putting a never-exiting process in a plain init container -> the pod hangs forever in `Init:` (an init must exit). If it should keep running, it's a sidecar.
- **Native sidecar on an old cluster** -> `restartPolicy` on an init container is only honored on ≥1.28 (gate) / ≥1.29 (GA). On older clusters that field is ignored or rejected; the helper won't behave as a sidecar.
